Thesis

The Local Thesis

The first wave of enterprise AI was cloud-first by default. Organizations sent their most sensitive data to third-party APIs, accepted opaque pricing, and traded control for convenience. That era is ending.

The next wave belongs to organizations that treat context as a durable asset. Portable, reusable context artifacts — prompt libraries, workflow specs, evaluation sets, and context libraries — survive each generation of models and tools. They compound in value instead of expiring with the next vendor pivot.

AI adoption is not an engineering-only problem. It spans operations, product, strategy, and culture. The common thread is architecture: local-first by default, cloud when justified, and never locked to a single provider. Models like Claude, GPT, and open-weight alternatives are interchangeable components — not dependencies.

Our thesis: the organizations that own their infrastructure and their context will outperform those that rent both. We exist to make that transition fast, safe, and repeatable.

Provider + Delivery Stack

High-level framework, low-level implementation

Context-first delivery is a stack problem: provider matrix, routing policy, agent execution, terminal workflows, and readout/observability.

Provider model plane

Text/coding, multimodal, image, speech, and retrieval models with explicit version tracking.

  • OpenAI, Claude, Gemini, Grok, Qwen, Mistral, Llama.
  • Updated from `provider-catalog/index.md` with live version pull.

Routing + aggregation plane

Fail-safe provider switching and policy-bound routing before workload reaches users.

  • OpenRouter-style abstraction, local policy layers, failover routing.
  • Fallbacks by function: coding, support, research, image, and retrieval.

Execution plane

Terminal-native and agent-native tools for production-level consistency.

  • Agents: Claude Code, Codex, Cursor, Windsurf, OpenHands.
  • Terminals: local shells, tmux, SSH, CI runners, containerized agents.

Readout plane

Quality, incidents, and model behavior are measured continuously.

  • Agent logs, evaluation traces, rollback checks, and issue summaries.
  • Small preview windows before broad rollout, then automated policy checks.

High-to-low offering model

One AI agency, two faces: consulting and free public academy

Every engagement ships the same architecture, from board-level outcomes to terminal-level execution. We do not separate services from teaching.

01Strategy and mandate

Define business outcomes, risk profile, data boundaries, and adoption targets before choosing tooling.

  • AI strategy, governance, and rollout model.
  • Operating model for owners, escalation, and accountability.
  • Public academy modules tied to the engagement scope.

02Context architecture

Context files become the durable asset across agents and model changes.

  • agents.md, settings.json, and domain entity notes.
  • Versioned workflows, decision logs, and project playbooks.
  • Low-friction handoff: markdown + one source of truth.

03Provider + model plane

Provider matrix and routing remain swappable.

  • Local/private deployment options and cloud fallback.
  • Text, coding, multimodal, image, speech, and retrieval models.
  • Version policy and failover before user-facing execution.

04Skill and workflow plane

High-level skills for portability; deterministic scripts for repeatability.

  • Skill-first design, then migration to scripts when stable.
  • Concise skill metadata, selective loading, and one-level references.
  • Use /skill-factory to standardize skill creation and evolution.

05Execution plane

Agent-native and terminal-native tooling with auditable outputs.

  • Claude Code, Codex, Cursor, Windsurf, OpenHands, etc.
  • Shell/terminal, tmux/SSH, CI runners, and secure environment control.
  • Repeatable CLI commands and structured handoff to humans.

06Readout and reliability

Behavior and value are measured at each layer, with controlled fallback.

  • Output quality checks, incident drills, rollback conditions.
  • Provider drift and catalog sync validation before release.
  • Decision logs to make updates safe and explainable.
