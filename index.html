Value flow in AI systems.

A ten-layer classification for the full AI stack — from silicon to products. When you discover a new tool, company, or startup, start here and ask: where does it live?

      Layers

      Everything in AI runs on matrix multiplications. The compute layer is the physical hardware and cloud infrastructure that executes those operations. GPU architecture, memory bandwidth, and interconnect topology determine what models can be trained, how fast inference runs, and what the floor cost is for any AI workload.

      The market is defined by NVIDIA's dominance in training (H100, B200, GB200 NVL72) and the emergence of alternatives for inference (AMD MI300X, Google TPU v5e, AWS Trainium2, Intel Gaudi 3). The key constraint is not raw FLOPS but memory bandwidth and interconnect — large models need to move weights between chips faster than they compute.

      Edge compute (Apple Neural Engine, Qualcomm Hexagon NPU) enables on-device inference for latency-sensitive or privacy-critical workloads. The gap between cloud and edge capability is shrinking — a MacBook Pro can run 70B parameter models locally via quantization.

      When to care about this layer: If you are training models from scratch, fine-tuning at scale, or running inference where latency or cost is the binding constraint. Most application builders never touch this layer directly — they consume it through Layer 04 (Inference) APIs.

      Pre-training is the phase where base models learn language, code, and world knowledge from massive datasets. This is the most capital-intensive part of the stack — training a frontier model costs $50M–$500M+ in compute alone. The result is a foundation model: a set of weights that encode statistical patterns over trillions of tokens.

      The inputs are data (web crawls, books, code, scientific papers), tokenizers (BPE, SentencePiece), and training infrastructure (distributed training frameworks like Megatron-LM, DeepSpeed, or custom stacks). The outputs are base models that can complete text but have no instruction-following ability — that comes in Layer 03.

      Data quality dominates model quality. The shift from "more data" to "better data" is the defining trend — Llama 3 was trained on 15T tokens of heavily filtered data, not just raw web crawl. Synthetic data generation (using existing models to create training data) is increasingly central to pre-training pipelines.

      When to care about this layer: Almost never directly. Unless you are a frontier lab or training a domain-specific base model from scratch. Most teams start at Layer 03 (fine-tuning an existing base model) or Layer 04 (using an API).

      Post-training turns a base model into something useful. It covers supervised fine-tuning (SFT), reinforcement learning from human feedback (RLHF), direct preference optimization (DPO), parameter-efficient methods (LoRA, QLoRA), knowledge distillation, and model merging. This is where a model learns to follow instructions, refuse harmful requests, and produce formatted outputs.

      The economics here are different from pre-training. Fine-tuning a model with LoRA can cost under $100 on a single GPU. Full fine-tuning of a 70B model might cost $5K–$50K. This layer is accessible to individual developers and small teams, making it the most democratized part of the training pipeline.

      Model merging (combining weights from multiple fine-tuned models without additional training) and distillation (training a smaller model to mimic a larger one) are increasingly important techniques. They let you create specialized models without the cost of full training runs.

      When to care about this layer: When API models do not perform well enough on your specific domain, when you need to reduce inference cost by using a smaller specialized model, or when you need to control model behavior beyond what prompting allows.

      Inference is running a trained model to produce outputs. This layer covers serving engines (the software that loads models and handles requests), cloud APIs (hosted endpoints), quantization (compressing models to run on smaller hardware), and the optimization techniques that make inference fast and cheap.

      The key metrics are tokens per second, time to first token (TTFT), cost per million tokens, and maximum concurrent requests. These vary by orders of magnitude between approaches — Groq's LPU hardware delivers 500+ tok/s, while a CPU-only llama.cpp setup might do 10 tok/s. The right choice depends on your latency requirements, volume, and budget.

      Quantization (GPTQ, AWQ, GGUF) reduces model size by 2-4x with minimal quality loss, enabling local inference on consumer hardware. Speculative decoding, continuous batching, and KV-cache optimization are the techniques that make serving engines competitive with each other.

      This layer is modality-agnostic. Text inference (LLMs), image generation (Stable Diffusion, DALL-E), video generation (Sora, Kling, Veo), text-to-speech (ElevenLabs, Bark, Coqui), and speech-to-text (Whisper, Deepgram, AssemblyAI) all live here. The models are trained at L02, but serving them to users is inference. The same cost/latency/throughput tradeoffs apply across modalities.

      When to care about this layer: Always. Every AI application consumes inference. The decision is: hosted API vs self-hosted, and if self-hosted, which engine and quantization format. Cost, latency, and data privacy are the three axes.

      Routing sits between your application and inference providers. It decides which model handles which request based on task type, cost constraints, latency requirements, and provider availability. Without a routing layer, you are hard-coding a single provider — and rebuilding when it changes pricing, goes down, or gets deprecated.

      The simplest routing is a fallback chain: try Provider A, if it fails try Provider B. More sophisticated routing classifies the incoming request (simple completion vs complex reasoning vs code generation) and dispatches to the best-fit model. Cost-aware routing tracks spend in real time and downgrades to cheaper models when budgets are tight.

      This layer is where model diversity becomes practical. Instead of picking one model for everything, you maintain a portfolio — cheap fast models for simple tasks, expensive capable models for hard ones, and local models for sensitive data. The router enforces this policy without the application code knowing which model it is talking to.

      When to care about this layer: As soon as you use more than one model or provider. Even with a single provider, a routing layer gives you fallback paths and the ability to switch without code changes.

      Orchestration is the framework layer that coordinates multi-step AI workflows. It covers agent frameworks, tool use, planning-execution-review loops, and the patterns for composing multiple model calls into reliable pipelines. This is where "prompt in, response out" becomes "task in, verified result out."

      The spectrum ranges from simple chain-of-calls (LangChain Expression Language, LlamaIndex pipelines) to autonomous agents that plan, execute tools, observe results, and iterate (Claude Code, OpenAI Codex CLI, AutoGen). The key design decision is how much autonomy to give the agent vs how much human oversight to require.

      Production orchestration needs retry logic, timeout handling, cost budgets per workflow, and structured handoffs between steps. Most framework demos skip these concerns — they matter enormously in production. The frameworks that survive are the ones that make reliability features first-class, not afterthoughts.

      When to care about this layer: When your use case involves more than a single prompt-response cycle. Multi-step workflows, tool use, human-in-the-loop approvals, and agent autonomy all live here.

      Context is everything the model sees beyond its training data. This layer covers retrieval-augmented generation (RAG), vector databases, embedding models, memory systems, and knowledge management tools. The quality of your context determines the quality of your outputs — a perfectly capable model with bad context produces bad results.

      RAG pipelines chunk documents into passages, embed them into vector space, store them in a vector database, and retrieve relevant chunks at query time. The engineering challenges are chunking strategy (too small loses context, too large dilutes relevance), embedding model selection, retrieval quality measurement, and keeping the index fresh as source documents change.

      Memory systems (conversation memory, agent memory, long-term knowledge) are the other half of this layer. They determine what the system remembers across sessions, what it forgets, and how it retrieves past interactions. CLAUDE.md files, memory plugins, and structured knowledge bases all live here.

      Where Obsidian fits: Obsidian, Notion, Logseq, and Roam are personal knowledge management tools that sit at this layer. Your notes become retrievable context — structured knowledge files (like CLAUDE.md), markdown documents for RAG pipelines, or direct input to agent memory systems. They are the human-authored knowledge layer that AI systems consume.

      When to care about this layer: Whenever the model needs information it was not trained on — your company docs, your codebase, your user data, your notes. RAG is the most common way to make AI systems useful for domain-specific work.

      Integrations connect AI systems to the tools and data sources they need to be useful. This includes protocol adapters (MCP — Model Context Protocol), coding agents embedded in IDEs, function calling interfaces, and connectors to external services (GitHub, Slack, databases, file systems, APIs).

      MCP is emerging as a standard protocol for tool integration — it defines how AI models discover, authenticate with, and call external tools. Before MCP, every tool integration was custom-built. MCP provides a shared interface that lets any model use any tool without provider-specific adapter code.

      Coding agents (Claude Code, Cursor, Windsurf, Continue, Aider) are the most mature integration category. They connect models to file systems, terminals, git, and language servers — turning a model that can reason about code into one that can write, test, and ship code. The quality of the integration determines how much context the model sees and how effectively it can take action.

      Eval and safety determine whether your AI system actually works and whether it fails safely. This layer covers evaluation frameworks (golden test sets, scoring rubrics, regression testing), observability (logging, tracing, cost tracking), guardrails (content filtering, output validation), and monitoring (quality dashboards, alerting).

      Without evals, you are guessing. Every prompt change, model upgrade, or context modification can silently degrade output quality. Eval sets are versioned test suites that catch regressions before they reach users. The best teams run evals in CI — a prompt change that degrades quality does not ship.

      Guardrails are runtime safety checks — they validate model outputs against rules before the output reaches the user. Content filters catch harmful outputs, format validators ensure structured outputs match schemas, and business logic checks enforce domain-specific constraints. They are the last line of defense between the model and the user.

      When to care about this layer: Before shipping anything to users. Evals should be the first thing you build, not the last. Guardrails are non-negotiable for user-facing applications. Observability is how you debug production issues.

      The product layer is where all the infrastructure below becomes useful to end users. Products are built on top of the other nine layers, combining inference, routing, orchestration, context, and integrations into something people actually use. This is the most crowded and fastest-moving layer.

      Product archetypes: Chatbots (ChatGPT, Claude, Gemini) — general-purpose conversational AI. Copilots (GitHub Copilot, Cursor, Windsurf) — AI assistants embedded in workflows. Agents (Devin, 11x, Bland AI) — autonomous AI that takes actions. Search (Perplexity, Glean) — AI-native information retrieval. Creative tools (Midjourney, Runway, Sora) — generative media. Vertical SaaS (Harvey, Abridge, Sierra) — domain-specific AI products.

      The defensibility question at this layer is critical. If your product is a thin wrapper over an API with no proprietary context, workflow, or data moat, you are competing on UX alone — and the model provider will likely build your feature natively. The products that survive have deep domain context, proprietary data, or workflow integration that cannot be replicated by a generic model.
