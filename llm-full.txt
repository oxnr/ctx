CTX POSITIONING
- AI transformation practice, not a traditional consultancy or agency.
- We help organizations adopt AI across engineering, operations, product, and strategy — with local-first architecture as the default.
- Every engagement includes a free, open academy layer. No separate academy product.
- Target: organizations that need sovereign, portable AI infrastructure — regulated environments, data-sensitive teams, and anyone who refuses vendor lock-in.
- Core approach: local-first deployment, portable context artifacts, broad AI adoption across functions.

THESIS
- The first wave of enterprise AI was cloud-first by default. Organizations traded control for convenience. That era is ending.
- Context is the durable asset. Portable, reusable context artifacts — prompt libraries, workflow specs, evaluation sets — survive each generation of models and tools.
- The organizations that own their infrastructure and their context will outperform those that rent both.
- AI adoption spans engineering, operations, product, strategy, and culture. The common thread is architecture: local-first by default, cloud when justified, never locked to a single provider.

PRINCIPLES
- Local-first by default: prefer on-premise and self-hosted deployment unless cloud is explicitly justified.
- Broad adoption surface: AI is not an engineering-only tool — design for every function.
- Context is the durable asset: deliver portable, reusable artifacts that survive tool generations.
- Start with high-impact use cases, then automate repeatable work.
- Everything is designed for reuse: prompts, skills, workflows, and context layers.
- Keep a local specialist agent library to reduce reinvented prompts and to preserve role-specific operating standards.
- Read less with each iteration: prioritize systems that make wrong outputs obvious and fixable quickly over exhaustive manual review.
- Keep planning at a chunk level: epics/decision blocks over overloaded ticket granularity, especially for fast-moving AI work.
- Treat issues as explicit context commitments, not a queue to become a bottleneck.
- Treat provider/model selection as a versioned infrastructure decision, not a preference.
- Keep a stable provider registry for context, so tool migrations are adapter changes, not methodology changes.
- Separate decision layers: provider sources, routing policies, terminal/agent tools, and governance/observability.
- Keep sessions contract-first: define scope, acceptance, and rollback before execution; split discovery and implementation phases.
- Use anti-sycophancy review loops for risky work: find assumptions, challenge them, then decide with minimal-rework rollback criteria.
- Treat context compaction/summarization as an operational optimization only. Keep canonical raw context where decisions can be audited, and pin compaction semantics/version in policy.
- Treat skills as measurable components, not only instruction bundles. New or changed skills should include a trigger test plan (positive + negative cases).

EXECUTION PRINCIPLES (How we run AI systems)
- We do not make code review a virtue. We make it a signal: small, high-confidence checks before larger releases.
- For every run, prove the change by outputs and contracts first, then review only what matters for risk.
- Design guardrails so teams can spend less time reading code and more time shaping architecture, intent, and assumptions.
- Build every stack in three layers:
  - provider + model layer
  - agent + workflow layer
  - terminal + delivery + observability layer

HARNESS-FIRST ENGINEERING PRINCIPLES
- Humans should optimize for design and control; agents execute implementation and iteration.
- The system should be legible to agents: repository-local, versioned, and self-contained.
- Start with minimal working scope and grow depth-first (goal -> decomposition -> next block).
- Keep boundaries explicit with contracts, not inferred assumptions.
- Build and test in a loop: execute -> validate -> recover -> retry.
- Treat docs like code: linked, versioned, and mechanically checkable.
- Keep a recurring cleanup loop for stale context, broken links, and old assumptions.
- Treat skills as controlled automation: track trigger quality and avoid accidental over-triggering.
- Start each run with context preload (`/recall`) before tool execution to reduce rework and reset drift.

REPEATED OPERATING LOOP
1. State intent and constraints in one short brief.
2. Define one narrow deliverable with one acceptance check.
3. Execute the relevant skill with complete context attachments.
4. Capture validation artifacts (results, logs, assumptions, risks).
5. Fix the root cause, not only the symptom.
6. Update decisions and plans before closing.
7. Add one cleanup action to reduce future agent ambiguity.
8. Include a minimal preview pass before escalation:
   - show a short proposed diff summary
   - highlight the highest-risk file/decision
   - require a stop/review gate only for high-impact failures
9. If a skill changed, include a quick trigger smoke check (true-positive and true-negative cases).
