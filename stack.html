Harness-first engineering.

Five layers, real tooling. Expand any plane for the specific tools, providers, and configurations we use.

System layers

Three surfaces: IDE, terminal, and web. Each surface has a primary agent runner and a defined handoff format. Claude Code runs in project mode with CLAUDE.md context files. Cursor uses .cursorrules for per-project behavior. Terminal sessions run through iTerm2 with tmux for multi-pane orchestration.

  • IDE agents: Claude Code (primary — project mode, tool use, file system access), Cursor (secondary — fast inline edits with .cursorrules), Windsurf (cascade mode for multi-file refactors). Each agent gets a scoped context file, not the full repo.
  • Terminal: iTerm2 + tmux sessions. One pane per agent task. Claude Code CLI for agentic terminal work. Warp for quick commands. Every terminal session is logged to ~/.claude/logs for audit replay.
  • Handoff format: JSON task objects — {task, context_capsule, acceptance_criteria, output_schema}. When a planning agent passes to a coding agent, the handoff is a typed contract, not a chat message.
  • Web surfaces: review dashboards, approval gates, and evaluation UIs. Built with plain HTML/CSS/JS — no framework dependency. Served locally or deployed to Cloudflare Pages.
  • Agent runners: Claude Code for autonomous multi-step work, Codex CLI for sandboxed code generation, OpenHands for browser-based agent tasks. Each runner maps to specific task types — never interchangeable.

Seven provider families behind one routing interface. Claude Sonnet 4 for complex reasoning and code. GPT-4o for fast structured outputs. Gemini 2.5 Pro for long-context analysis. Llama 3.1 70B via Ollama for local-only tasks. Every model is pinned to a specific version — no silent upgrades.

  • Anthropic: Claude Opus 4 (complex reasoning, architecture), Claude Sonnet 4 (primary workhorse — code, analysis, review), Claude Haiku 3.5 (classification, routing, formatting at $0.25/MTok input).
  • OpenAI: GPT-4o (structured JSON outputs, function calling), GPT-4o-mini (fast low-cost tasks). o3 for mathematical/logical reasoning tasks that need chain-of-thought verification.
  • Google: Gemini 2.5 Pro (1M token context — full codebase analysis, long document processing), Gemini 2.0 Flash (vision tasks, image analysis, multimodal routing).
  • Local models: Llama 3.1 70B and Qwen 2.5 72B via Ollama for sensitive data that cannot leave the machine. Mistral Large for European data residency requirements. vLLM for production-grade local serving.
  • Embedding: text-embedding-3-large (OpenAI) for semantic search, nomic-embed-text (local) for private data. All embeddings stored in pgvector with cosine similarity indexing.
  • Cost tracking: every API call logged with model, tokens, latency, and cost. Claude Sonnet: ~$3/MTok input, $15/MTok output. Haiku: ~$0.25/$1.25. Budget alerts at 80% of per-team monthly limits.

OpenRouter for multi-provider API access with automatic failover. LiteLLM as self-hosted proxy for teams that need full control. Custom routing policies per task type — simple formatting goes to Haiku, code generation to Sonnet, sensitive data to local Llama. Every routing decision is logged.

  • Task-type routing table: classification → Haiku, code generation → Sonnet, architecture review → Opus, summarization → GPT-4o-mini, long-doc analysis → Gemini 2.5 Pro, sensitive PII → local Llama 3.1. The table is a versioned config file, not hardcoded.
  • OpenRouter: primary gateway for cloud models. Automatic provider failover — if Anthropic is rate-limited, reroute to backup. Usage tracked per-key, per-team. Spend limits enforced at the gateway.
  • LiteLLM proxy: self-hosted on internal infrastructure for teams with data residency requirements. Same API interface as OpenRouter — swap by changing one environment variable. Supports custom model aliases.
  • Risk classification: customer data → local only. Financial outputs → Opus + human review. Public content → any model, fast routing. Internal docs → Sonnet with standard context. Risk level determines eligible models and review requirements.
  • Fallback chains: Sonnet → GPT-4o → local Llama (degraded). Each fallback is pre-tested with golden sets. Fallback triggers on: rate limit, latency > 30s, error rate > 5%, or provider outage. Notification fires on every fallback activation.
  • Audit: every routing decision → {timestamp, task_type, model_selected, alternatives_considered, latency_ms, cost_usd, outcome}. Queryable via SQL. Weekly reports per team.

Agents execute in sandboxed environments with explicit permission boundaries. Claude Code gets file system read/write within the project directory. No network access unless explicitly granted per task. Every tool call is logged with input, output, duration, and exit code.

  • Sandbox model: Docker containers for untrusted code execution. Claude Code project mode for trusted repo work — permissions defined in CLAUDE.md (allowed commands, file patterns, network rules). Codex CLI runs in isolated sandbox by default.
  • Permission boundaries: file system (read/write per directory pattern), network (allowlist per domain), process (allowed commands list), secrets (env vars injected per task, never in context). Violations logged and blocked.
  • CI integration: GitHub Actions for automated testing of agent outputs. Every PR from an agent runs the same pipeline as human PRs — lint, type-check, test, build. Agent PRs get auto-labeled for review priority.
  • Tool contracts: each tool has a typed schema — input (JSON Schema), output (JSON Schema), permissions (file/network/process), timeout (seconds). Tools are registered in a catalog. Agents discover tools from the catalog, not from prompts.
  • Audit trail: {timestamp, agent_id, tool_name, input_hash, output_hash, duration_ms, exit_code, permissions_used}. Stored in append-only log. Queryable for debugging, compliance, and cost attribution. 90-day retention default.

The durable layer. CLAUDE.md files as behavioral contracts per project. Context capsules stored as versioned markdown in .ctx/ directories. Decision logs in append-only files. Session memory in SQLite, promoted to durable context via explicit commit. Everything here survives model swaps and provider changes.

  • Context capsules: markdown files in .ctx/capsules/ — one per domain (architecture, decisions, evaluation, team). Each capsule has a header with scope, last-updated, and expiry. Capsules are loaded into agent context at session start based on task type.
  • CLAUDE.md contracts: per-project behavioral definitions — coding standards, allowed patterns, forbidden patterns, review criteria, test requirements. The agent reads this file first. It is the behavioral boundary for everything that follows.
  • Decision log: .ctx/decisions/ — append-only markdown. Each entry: {date, decision, rationale, alternatives_considered, owner}. Agents reference decisions before proposing changes. Contradicting a logged decision requires explicit override.
  • Session memory: SQLite database at .ctx/memory.db. Current session state, scratch work, intermediate results. Volatile by default. Promoted to capsule via `ctx promote` command when the team decides something is durable.
  • Recovery: every context mutation is a git commit in the .ctx/ directory. Rollback to any previous state with `git log .ctx/`. Corruption recovery: rebuild capsules from decision log + source code analysis. Tested quarterly.
