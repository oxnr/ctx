Classification

Layer 01 — Compute. GPUs, TPUs, custom accelerators, edge chips, and GPU cloud providers like NVIDIA, AMD, Cerebras, CoreWeave, and Lambda. Cloud providers that resell GPU capacity (AWS, GCP) are at this layer when they sell raw compute, but sit at L04 Inference or L05 Routing when they offer model-serving APIs. The distinction is whether they are selling hardware access or model access.

Layer 02 — Pre-Training. This is frontier labs (OpenAI, Anthropic, Google DeepMind, Meta, Mistral, DeepSeek) and training data companies (Scale AI, FineWeb). If a company trains a base model from raw data to initial weights, it is pre-training. If it takes an existing base model and adapts it, that is Layer 03.

Layer 03 — Post-Training. LoRA platforms, RLHF tools, distillation frameworks, model merging. Companies like Unsloth, Predibase, and Axolotl live here. The dividing line with L02: pre-training creates a new base model from data; post-training adapts an existing one.

Layer 04 — Inference. This is the largest category by revenue. Serving engines (vLLM, llama.cpp, TGI), cloud APIs (OpenAI API, Anthropic API, Vertex AI, Bedrock), local runners (Ollama, LM Studio), and quantization tools. If you send a prompt and get a completion back, the thing handling that request is inference infrastructure.

Layer 05 — Routing. Model routers, gateways, proxy layers, cost optimizers. OpenRouter, LiteLLM, Portkey, and Martian are routing tools. If a tool sits between your application and the inference provider, deciding which model gets the request, it is routing. A common edge case: Helicone does routing AND observability — it is classified in both L05 and L09.

Layer 06 — Orchestration. Agent frameworks (LangChain, CrewAI, AutoGen), coding agents (Claude Code, Codex CLI, Devin), and workflow builders (Dify, n8n). If the tool manages a loop of plan-execute-review or coordinates multiple model calls, it is orchestration. Single-call wrappers are not orchestration — they are just inference with a prompt.

Layer 07 — Context and Knowledge. Vector databases (Pinecone, Weaviate, Chroma, Qdrant), embedding providers (Voyage AI, Jina), knowledge management tools (Obsidian, Notion), and memory systems. If it helps a model access information beyond its training data, it is context infrastructure. RAG pipelines span this layer.

Layer 08 — Integrations. MCP servers, coding assistants embedded in IDEs (Cursor, Windsurf, Continue), API connectors (Composio, Zapier AI Actions), and tool-use frameworks. The key question: does it extend what the model can do by connecting it to external systems? If yes, it is an integration.

Layer 09 — Eval and Safety. Eval frameworks (Braintrust, Promptfoo, LangSmith), observability (Langfuse, Helicone), guardrails (Guardrails AI, NeMo Guardrails, Llama Guard), and cost tracking. If the tool watches, measures, validates, or filters AI outputs, it is eval and safety.

Layer 10 — Products. Chatbots (ChatGPT, Claude, Gemini), copilots (GitHub Copilot, Cursor), agents (Devin, 11x, Sierra), search (Perplexity), creative tools (Midjourney, Runway), and vertical SaaS (Harvey, Abridge). If a human interacts with AI through it, it is a product. Sub-categories: chatbot, copilot, agent, search, creative, platform, vertical.

About the value chain

The classification emerged from mapping roughly 260 tools and identifying where meaningful boundaries exist. Ten is the minimum number that keeps categories meaningfully distinct. Fewer layers would force unlike things together — routing is fundamentally different from orchestration, and context management is different from integrations. More layers would create artificial splits without adding classification power.

Many tools span layers. We classify by primary function (direct layers) and secondary presence (indirect layers). Click any tool in the explorer on the main page to see both. For example, LangChain is primarily orchestration (L06) but has indirect presence in context (L07) and integrations (L08). The primary layer is where the tool adds the most distinctive value.

No. Other frameworks exist — the Sequoia AI stack, the a16z infrastructure map, and various analyst reports all organize the space differently. This one optimizes for practitioners who need to understand where a tool fits in a production system. The ten layers follow the actual flow of value from hardware to user-facing products.

Products (L10) have sub-categories that describe the user interaction pattern: chatbot (conversational interface), copilot (embedded assistant), agent (autonomous task executor), search (information retrieval), creative (generative media), platform (horizontal tooling), and vertical (domain-specific application). These describe how the user interacts, not the underlying technology.

About ctx

An open knowledge base for teams building with AI. Maps the full value chain from silicon to products, with a tool explorer, learning modules, and a glossary. Everything is plain HTML hosted on GitHub Pages — no framework, no build step.
